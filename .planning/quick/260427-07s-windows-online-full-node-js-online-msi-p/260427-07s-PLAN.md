# Quick Task 260427-07s Plan

## Goal

梳理并修复 Windows online/full 两条 Node.js 安装链路：online 版本负责下载
MSI、展示进度、刷新当前进程可见 PATH，并在安装完成后立刻保证程序内 npm
可用；full 版本继续使用 `assets-windows` 中的 MSI 打包到资源目录，运行时直接
发现并安装，同样保证安装后应用内的 npm/pnpm 调用可用。

## Context

- 上一个 quick task `260426-wph` 已统一 full 包资源扫描路径，并把 workflow 收敛到
  `src-tauri/resources/nodejs`。
- 当前 `install_node_env` 已能区分 bundled/online MSI，但“安装成功”并不等于“当前
  应用进程中的 npm 立刻可调用”；`set_npm_registry`、`install_pnpm`、`run_project_task`
  以及环境检测仍有依赖旧 PATH 或直接调 `npm`/`pnpm` 的风险。
- 本次 quick task 不引入新依赖、不做 research，只在现有 Tauri Rust 与 release
  workflow 约定内把 Windows 安装链路补齐。

## Tasks

### Task 1
- files: `src-tauri/src/lib.rs`
- action: 抽取一组共享的 Windows Node.js 可执行定位 helper，统一返回已安装 Node 目录、`node.exe`、`npm.cmd`、`pnpm.cmd` 的优先路径，并能把该目录注入当前命令执行环境。把 `check_node_env`、`set_npm_registry`、`install_pnpm`、`run_project_task` 这些会在应用内直接调用 Node/npm/pnpm 的逻辑全部改为优先使用该 helper，而不是分别硬编码 `C:\Program Files\nodejs` 或裸调 `npm`/`pnpm`。重点是解决“MSI 已安装成功，但当前应用进程里 npm 还不可用”的问题；不要改动非 Node 环境相关命令。
- verify: `cargo check --manifest-path src-tauri/Cargo.toml`
- done: Windows 下与 Node 环境相关的 Rust 命令共享同一套路径解析与 PATH 注入逻辑，安装后无需重启应用即可在程序内命中正确的 `npm.cmd`。

### Task 2
- files: `src-tauri/src/lib.rs`
- action: 重写 `install_node_env` 的 Windows 收尾逻辑，让 online/full 两条路径都在 MSI 安装完成后执行同一套“环境就绪验证”：刷新进程 PATH、重新解析 Node 目录、用显式命中的 `npm.cmd` 做一次轻量探测（例如 `npm -v` 或 `npm config get registry`），只有探测通过才返回成功。保留 online 下载进度事件，并补足 bundled/full 路径的阶段性进度文案（发现内置 MSI、复制、安装、校验可用性）。若 MSI 安装完成但 npm 探测失败，错误信息要明确指出是“安装后程序内 npm 不可用”，并携带探测的可执行路径。不要把 full 路径偷偷回退为 online 下载。
- verify: `cargo check --manifest-path src-tauri/Cargo.toml && cargo test --manifest-path src-tauri/Cargo.toml --lib`
- done: Windows online/full 两条安装链路都以“程序内 npm 可调用”作为成功标准；用户能看到下载/安装/校验进度，失败时能区分资源缺失、MSI 失败、npm 不可用。

### Task 3
- files: `.github/workflows/release.yml`, `src-tauri/src/lib.rs`
- action: 固化 Windows full 打包契约，继续以 `src-tauri/assets-windows/{MSI}` 为单一资产来源，构建前校验 MSI 存在并复制到 `src-tauri/resources/nodejs`，同时让运行时扫描与 Task 2 的 full 安装文案都明确围绕这一目录展开。workflow 不要引入新的下载步骤，运行时代码也不要硬编码具体 MSI 版本号；只要求 workflow 保证资源路径正确、运行时按 `.msi` 发现。必要时在计划内顺手清理任何会让扫描目录与 workflow 目录再次漂移的重复常量。
- verify: `node -e "const fs=require('fs');const y=fs.readFileSync('.github/workflows/release.yml','utf8');if(!y.includes('assets-windows')||!y.includes('resources/nodejs'))process.exit(1);" && cargo check --manifest-path src-tauri/Cargo.toml`
- done: full 版本从 `assets-windows` → `resources/nodejs` → 运行时 MSI 发现/安装 是一条单一契约链路，不会出现“包里有 MSI 但应用找不到”或“找到 MSI 但安装后程序内 npm 不可用”。

## Threat model

### Trust boundaries

| Boundary | Description |
|----------|-------------|
| Internet download → local MSI execution | online 版本会下载并执行 MSI，必须把下载、安装、安装后可用性校验分层诊断 |
| GitHub Actions bundle resources → runtime resource scan | full 版本依赖 CI 把 MSI 放进约定目录，路径漂移会让运行时误判资源缺失 |
| Installed Node directory → in-app command execution | 即使 MSI 安装成功，若当前进程 PATH/命令解析未刷新，程序内 npm 仍会失败 |

### STRIDE threat register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-quick-01 | D | `install_node_env` Windows online flow | mitigate | 把下载、MSI 安装、npm 就绪探测拆成明确阶段并输出进度/错误，避免 silent failure |
| T-quick-02 | T | Windows full bundle resource contract | mitigate | workflow 只从 `assets-windows` 复制到 `resources/nodejs`，运行时只按约定目录扫描 `.msi` |
| T-quick-03 | D | in-app npm execution after install | mitigate | 共享 Windows node/npm 解析 helper，安装后刷新 PATH 并用显式 `npm.cmd` 做 readiness probe |

## Success criteria

- Windows online 安装完成后，当前应用进程中的 `npm` 相关命令无需重启即可可用。
- Windows full 包继续直接使用内置 MSI 安装，不回退为在线下载。
- 下载进度、安装进度、安装后可用性校验都有清晰事件或错误信息。
- 改动范围仅限 Windows Node.js 安装链路与 full 打包契约。
